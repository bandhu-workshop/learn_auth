## 2. Authentication & Transport

### Access Token

- **Format:** Signed JWT (HS256)
- **Lifetime:** 15 minutes (configurable via `JWT_ACCESS_TOKEN_EXPIRE_MINUTES`)
- **Transport:** `Authorization` request header — Bearer scheme
- **Client storage:** In-memory only (never `localStorage` — XSS risk)

```
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

**JWT payload claims:**

| Claim | Type | Description |
|---|---|---|
| `sub` | `str` | User ID as string |
| `email` | `str` | User email |
| `role` | `str` | User role (`"user"` or `"admin"`) |
| `type` | `str` | Always `"access"` — distinguishes from refresh tokens |
| `iat` | `int` | Issued-at Unix timestamp |
| `exp` | `int` | Expiry Unix timestamp |

---

### Refresh Token

- **Format:** Signed JWT (HS256)
- **Lifetime:** 7 days (configurable via `JWT_REFRESH_TOKEN_EXPIRE_DAYS`)
- **Transport:** HttpOnly cookie — **never returned in the JSON body**
- **Cookie attributes:**

| Attribute | Value | Reason |
|---|---|---|
| `HttpOnly` | `true` | JavaScript cannot read it — blocks XSS token theft |
| `Secure` | `true` | Transmitted over HTTPS only |
| `SameSite` | `Strict` | Cookie not sent on cross-site requests — blocks CSRF |
| `Path` | `/api/v1/auth` | Cookie scoped to auth endpoints only |
| `Max-Age` | `604800` (7 days) | Matches token expiry |

**Cookie set by server (on login / refresh):**
```
Set-Cookie: refresh_token=eyJ...; HttpOnly; Secure; SameSite=Strict; Path=/api/v1/auth; Max-Age=604800
```

**Cookie cleared by server (on logout):**
```
Set-Cookie: refresh_token=; HttpOnly; Secure; SameSite=Strict; Path=/api/v1/auth; Max-Age=0
```

## 3. Error Format

All errors use FastAPI's `HTTPException` envelope:

```json
{
  "detail": "Human-readable error message."
}
```

Pydantic validation errors (`422`) use a richer shape auto-generated by FastAPI:

```json
{
  "detail": [
    {
      "loc": ["body", "password"],
      "msg": "Password must contain at least one digit",
      "type": "value_error"
    }
  ]
}
```

### HTTP Status Code Reference

| Code | Name | When used |
|---|---|---|
| `200` | OK | Successful GET / POST with response body |
| `201` | Created | Resource successfully created |
| `204` | No Content | Successful action with no response body |
| `400` | Bad Request | Business rule violation (wrong password, same-as-old password, etc.) |
| `401` | Unauthorized | Missing, expired, or invalid token or credentials |
| `403` | Forbidden | Authenticated but not permitted (deactivated account, wrong role) |
| `409` | Conflict | Uniqueness violation (email already registered) |
| `422` | Unprocessable Entity | Pydantic field validation failure (auto-generated by FastAPI) |

## 4. Field Validation Rules

These rules are the source of truth for Pydantic schema validators.
They are referenced individually by each endpoint below.

### Rule: `email`

| Rule | Detail |
|---|---|
| Format | Valid RFC 5321 email (`pydantic.EmailStr`) |
| Normalisation | Lowercased automatically by Pydantic |
| Max length | 255 characters — enforced in both DB column and Pydantic `Field(max_length=255)` |

### Rule: `password` (on input — register and change-password only)

| Rule | Detail |
|---|---|
| Min length | 8 characters — `Field(min_length=8)` |
| Max length | 128 characters — `Field(max_length=128)` |
| Uppercase | At least one `[A-Z]` character — `re.search(r"[A-Z]", v)` |
| Lowercase | At least one `[a-z]` character — `re.search(r"[a-z]", v)` |
| Digit | At least one `[0-9]` character — `re.search(r"\d", v)` |
| Validator location | Pydantic `@field_validator` on `RegisterRequest` and `ChangePasswordRequest` |

> Login does **not** apply strength rules — the password is passed directly to `verify_password()`.

### Rule: `full_name`

| Rule | Detail |
|---|---|
| Type | `str` or `null` |
| Max length | 255 characters |
| Required | No |

### Rule: `avatar_url`

| Rule | Detail |
|---|---|
| Type | `str` or `null` |
| Max length | 512 characters |
| Required | No |

### Rule: `bio`

| Rule | Detail |
|---|---|
| Type | `str` or `null` |
| Max length | 1000 characters |
| Required | No |

## 5. Endpoints — Core Auth

Base prefix: `/api/v1/auth`

---

### POST /auth/register

Create a new user account.

**Authentication required:** No  
**Request Content-Type:** `application/json`

---

#### Request Body

```json
{
  "email": "alice@example.com",
  "password": "SecurePass1",
  "full_name": "Alice Smith"
}
```

| Field | Type | Required | Validation | Description |
|---|---|---|---|---|
| `email` | `string` | **Yes** | Valid RFC 5321 email, max 255 chars | Login identifier. Normalised to lowercase before storage. |
| `password` | `string` | **Yes** | 8–128 chars; ≥1 uppercase, ≥1 lowercase, ≥1 digit | Plaintext. Hashed with Argon2id server-side. Never logged or returned. |
| `full_name` | `string \| null` | No | Max 255 chars | Optional display name. Stored as-is. |

---

#### Success Response — `201 Created`

```json
{
  "id": 42,
  "email": "alice@example.com",
  "full_name": "Alice Smith",
  "avatar_url": null,
  "bio": null,
  "is_verified": false,
  "is_active": true,
  "role": "user",
  "created_at": "2026-03-13T10:30:00Z"
}
```

| Field | Type | Nullable | Description |
|---|---|---|---|
| `id` | `integer` | No | Auto-incremented primary key. |
| `email` | `string` | No | Lowercased email address. |
| `full_name` | `string` | Yes | Display name; `null` if not provided at registration. |
| `avatar_url` | `string` | Yes | Always `null` at registration. |
| `bio` | `string` | Yes | Always `null` at registration. |
| `is_verified` | `boolean` | No | Always `false` at registration — email not yet verified. |
| `is_active` | `boolean` | No | Always `true` at registration. |
| `role` | `string` | No | Always `"user"` at registration. Possible values: `"user"`, `"admin"`. |
| `created_at` | `string (datetime)` | No | UTC ISO 8601 timestamp of account creation. |

---

#### Error Responses

| Status | `detail` | Condition |
|---|---|---|
| `409 Conflict` | `"Email already registered"` | An account with this email already exists, including soft-deleted accounts. |
| `422 Unprocessable Entity` | *(Pydantic field errors array)* | Any field fails validation — see Section 4. |

---

#### Side Effects

- A new `User` row is inserted with `is_verified=false`, `is_active=true`, `role="user"`.
- Password is hashed with Argon2id before storage. The plaintext is never persisted.

---

### POST /auth/login

Authenticate a user and issue an access token plus a refresh token cookie.

**Authentication required:** No  
**Request Content-Type:** `application/json`

---

#### Request Body

```json
{
  "email": "alice@example.com",
  "password": "SecurePass1"
}
```

| Field | Type | Required | Validation | Description |
|---|---|---|---|---|
| `email` | `string` | **Yes** | Valid RFC 5321 email | The registered email address to look up. |
| `password` | `string` | **Yes** | None (no strength check on login) | Plaintext password to verify against the stored Argon2id hash. |

---

#### Success Response — `200 OK`

Access token in body; refresh token set as HttpOnly cookie.

```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "bearer"
}
```

| Field | Type | Nullable | Description |
|---|---|---|---|
| `access_token` | `string` | No | Short-lived JWT (15 min). Client must store in memory — never in `localStorage`. |
| `token_type` | `string` | No | Always `"bearer"`. |

Response also sets the refresh token cookie:
```
Set-Cookie: refresh_token=<jwt>; HttpOnly; Secure; SameSite=Strict; Path=/api/v1/auth; Max-Age=604800
```

---

#### Error Responses

| Status | `detail` | Condition |
|---|---|---|
| `401 Unauthorized` | `"Invalid credentials"` | Email not found **or** password incorrect. Same message for both — prevents user enumeration. Response time is always identical (timing-safe via dummy hash). |
| `401 Unauthorized` | `"Account is locked. Try again after 2026-03-13T10:45:00Z"` | Account locked due to too many failed attempts. The `locked_until` timestamp is embedded in the message. |
| `403 Forbidden` | `"Account is deactivated"` | `is_active = false` on the user record. |

---

#### Side Effects

On **success:**
- `failed_login_attempts` reset to `0` on the `User` row.
- `is_locked` reset to `false`, `locked_until` cleared.
- A new `RefreshToken` row is created in the DB with a fresh `family_id` (UUID), `jti` (UUID), `token_hash`, and `expires_at`.

On **failure:**
- `failed_login_attempts` incremented by `1`.
- When `failed_login_attempts >= MAX_FAILED_LOGIN_ATTEMPTS` (default: 5): `is_locked = true`, `locked_until = now() + ACCOUNT_LOCKOUT_MINUTES` (default: 15 min).

---

### POST /auth/refresh

Exchange a valid refresh token cookie for a new access token. The refresh token is rotated on every successful call.

**Authentication required:** No — refresh token is read from the `refresh_token` HttpOnly cookie.  
**Request Content-Type:** None

---

#### Request Body

None.

---

#### Success Response — `200 OK`

```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "bearer"
}
```

| Field | Type | Nullable | Description |
|---|---|---|---|
| `access_token` | `string` | No | New short-lived JWT. Discard the previous one. |
| `token_type` | `string` | No | Always `"bearer"`. |

Response also sets a **new rotated** refresh token cookie:
```
Set-Cookie: refresh_token=<new_jwt>; HttpOnly; Secure; SameSite=Strict; Path=/api/v1/auth; Max-Age=604800
```

---

#### Error Responses

| Status | `detail` | Condition |
|---|---|---|
| `401 Unauthorized` | `"Refresh token missing"` | The `refresh_token` cookie is absent from the request. |
| `401 Unauthorized` | `"Invalid or expired refresh token"` | JWT signature is invalid, token is malformed, or `exp` is in the past. |
| `401 Unauthorized` | `"Refresh token has been revoked"` | JWT decoded successfully; `jti` found in DB but `revoked_at` is set (already logged out). |
| `401 Unauthorized` | `"Token reuse detected — all sessions revoked"` | `jti` not found in DB at all — it was already rotated away, indicating the token was stolen and replayed. Entire `family_id` is revoked. |

---

#### Side Effects

On **success:**
- Old `RefreshToken` DB row: `revoked_at = now()`.
- New `RefreshToken` DB row created: same `family_id`, new `jti`, new `token_hash`, new `expires_at`.

On **reuse detection (theft):**
- All `RefreshToken` rows matching the stolen token's `family_id` are set `revoked_at = now()`.
- The user must log in again fresh.

---

### POST /auth/logout

Revoke the current session's refresh token and clear the cookie.

**Authentication required:** Bearer token preferred; cookie alone is sufficient.  
**Request Content-Type:** None

---

#### Request Body

None.

---

#### Success Response — `204 No Content`

No body. Always succeeds, even if the cookie was absent (idempotent by design).

Response clears the cookie:
```
Set-Cookie: refresh_token=; HttpOnly; Secure; SameSite=Strict; Path=/api/v1/auth; Max-Age=0
```

---

#### Error Responses

None. Logout never returns an error.

---

#### Side Effects

- The `RefreshToken` row for the current `jti` is marked `revoked_at = now()`.
- If no matching token is found in the DB (cookie absent or already revoked), no error is raised — the response is still `204`.

---

### POST /auth/logout-all

Revoke all refresh tokens for the authenticated user across every device and session.

**Authentication required:** Bearer token (required — must identify who to log out).  
**Request Content-Type:** None

---

#### Request Body

None.

---

#### Success Response — `204 No Content`

No body.

Response also clears the current cookie:
```
Set-Cookie: refresh_token=; HttpOnly; Secure; SameSite=Strict; Path=/api/v1/auth; Max-Age=0
```

---

#### Error Responses

| Status | `detail` | Condition |
|---|---|---|
| `401 Unauthorized` | `"Not authenticated"` | Bearer token missing or expired. |

---

#### Side Effects

- All `RefreshToken` rows where `user_id = <current user>` and `revoked_at IS NULL` are updated: `revoked_at = now()`.
- Current session cookie is cleared.

### PATCH /users/me/profile

Update mutable profile fields. **PATCH semantics** — only fields present in the request body are changed.

**Authentication required:** Bearer token  
**Request Content-Type:** `application/json`

---

#### Request Body

All fields are optional. Send only the fields you want to change.

```json
{
  "full_name": "Alice J. Smith",
  "bio": "Backend developer, now with 20% more coffee."
}
```

| Field | Type | Required | Validation | Description |
|---|---|---|---|---|
| `full_name` | `string \| null` | No | Max 255 chars | Omit to leave unchanged. Send `null` to clear (set DB to `NULL`). |
| `avatar_url` | `string \| null` | No | Max 512 chars | Omit to leave unchanged. Send `null` to clear. |
| `bio` | `string \| null` | No | Max 1000 chars | Omit to leave unchanged. Send `null` to clear. |

**PATCH semantics — field behaviour matrix:**

| What is sent | Effect on the DB column |
|---|---|
| Field **omitted** entirely | DB value is **unchanged** |
| `"field": null` | DB value **set to `NULL`** — clears the field |
| `"field": "some value"` | DB value **set to `"some value"`** |

Enforced via `model_dump(exclude_unset=True)` in the service layer.

---

#### Success Response — `200 OK`

Returns the full updated user object (same shape as `GET /users/me`).

```json
{
  "id": 42,
  "email": "alice@example.com",
  "full_name": "Alice J. Smith",
  "avatar_url": "https://example.com/avatars/alice.jpg",
  "bio": "Backend developer, now with 20% more coffee.",
  "is_verified": true,
  "is_active": true,
  "role": "user",
  "created_at": "2026-03-13T10:30:00Z"
}
```

Same field definitions as `GET /users/me`.

| Status | `detail` | Condition |
|---|---|---|
| `401 Unauthorized` | `"Not authenticated"` | Bearer token missing or expired. |
| `422 Unprocessable Entity` | *(Pydantic field errors)* | Field constraint violation — e.g., `bio` exceeds 1000 chars. |

### PUT /users/me/password

Change the authenticated user's password. Current password must be provided for verification.

**Authentication required:** Bearer token  
**Request Content-Type:** `application/json`

---

#### Request Body

```json
{
  "current_password": "OldSecurePass1",
  "new_password": "NewSecurePass2"
}
```

| Field | Type | Required | Validation | Description |
|---|---|---|---|---|
| `current_password` | `string` | **Yes** | None | Must match the user's currently stored Argon2id hash. Provides proof of identity before allowing the change. |
| `new_password` | `string` | **Yes** | 8–128 chars; ≥1 uppercase, ≥1 lowercase, ≥1 digit | The replacement password. Must differ from `current_password`. Hashed Argon2id before storage. |

---

#### Success Response — `204 No Content`

No body.

---

#### Error Responses

| Status | `detail` | Condition |
|---|---|---|
| `400 Bad Request` | `"Current password is incorrect"` | `current_password` does not match the Argon2id hash in the DB. |
| `400 Bad Request` | `"New password must be different from current password"` | `new_password` is identical to `current_password` — checked with plain-text comparison before hashing. |
| `401 Unauthorized` | `"Not authenticated"` | Bearer token missing or expired. |
| `422 Unprocessable Entity` | *(Pydantic field errors)* | `new_password` fails length or strength validation. |

---

#### Side Effects

- `User.hashed_password` updated to new Argon2id hash.
- **All `RefreshToken` rows for this user are revoked** (`revoked_at = now()`). Every other active session (other devices) must re-authenticate.
- The current Bearer access token remains valid until its natural expiry (~15 min) — stateless tokens cannot be individually revoked.

---

### DELETE /users/me

Soft-delete the authenticated user's account. Requires password confirmation.

**Authentication required:** Bearer token  
**Request Content-Type:** `application/json`

---

#### Request Body

```json
{
  "password": "SecurePass1"
}
```

| Field | Type | Required | Validation | Description |
|---|---|---|---|---|
| `password` | `string` | **Yes** | None | The user's current password. Required as an explicit confirmation step to prevent accidental deletion. |

---

#### Success Response — `204 No Content`

No body.

Response also clears the refresh token cookie:
```
Set-Cookie: refresh_token=; HttpOnly; Secure; SameSite=Strict; Path=/api/v1/auth; Max-Age=0
```

---

#### Error Responses

| Status | `detail` | Condition |
|---|---|---|
| `400 Bad Request` | `"Password confirmation failed"` | Provided password does not match the stored Argon2id hash. |
| `401 Unauthorized` | `"Not authenticated"` | Bearer token missing or expired. |

---

#### Side Effects

- `User.deleted_at` set to `now()`.
- All `RefreshToken` rows for this user revoked immediately (`revoked_at = now()`).
- Refresh token cookie cleared in the response.
- Account becomes inaccessible — login attempts with this email return `401 "Invalid credentials"`.
- The email address remains "taken" — re-registration with the same email is blocked.
- The `User` row is **never physically removed** — preserved for audit trail and referential integrity.

## 7. Pydantic Schema Blueprints

These are the exact schemas to implement in `src/learn_auth/app/schemas/auth.py`.
Every field, validator, and config option is specified — implement exactly as written.

```python
# src/learn_auth/app/schemas/auth.py
import re
from datetime import datetime

from pydantic import BaseModel, ConfigDict, EmailStr, Field, field_validator


# ---------------------------------------------------------------------------
# Shared validator — used by RegisterRequest and ChangePasswordRequest
# ---------------------------------------------------------------------------

def _validate_password_strength(v: str) -> str:
    if not re.search(r"[A-Z]", v):
        raise ValueError("Password must contain at least one uppercase letter")
    if not re.search(r"[a-z]", v):
        raise ValueError("Password must contain at least one lowercase letter")
    if not re.search(r"\d", v):
        raise ValueError("Password must contain at least one digit")
    return v


# ---------------------------------------------------------------------------
# Request schemas
# ---------------------------------------------------------------------------

class RegisterRequest(BaseModel):
    email: EmailStr                                       # normalised to lowercase by Pydantic
    password: str = Field(min_length=8, max_length=128)
    full_name: str | None = Field(default=None, max_length=255)

    @field_validator("password")
    @classmethod
    def password_strength(cls, v: str) -> str:
        return _validate_password_strength(v)


class LoginRequest(BaseModel):
    email: EmailStr
    password: str    # no strength check — just passed to verify_password()


class UpdateProfileRequest(BaseModel):
    full_name: str | None = Field(default=None, max_length=255)
    avatar_url: str | None = Field(default=None, max_length=512)
    bio: str | None = Field(default=None, max_length=1000)
    # All fields optional — empty body {} is valid and performs no update


class ChangePasswordRequest(BaseModel):
    current_password: str
    new_password: str = Field(min_length=8, max_length=128)

    @field_validator("new_password")
    @classmethod
    def password_strength(cls, v: str) -> str:
        return _validate_password_strength(v)


class DeleteAccountRequest(BaseModel):
    password: str    # confirmation only — no strength check


# ---------------------------------------------------------------------------
# Response schemas
# ---------------------------------------------------------------------------

class TokenResponse(BaseModel):
    access_token: str
    token_type: str = "bearer"
    # refresh_token is NOT here — it is delivered via Set-Cookie header only


class UserResponse(BaseModel):
    id: int
    email: str
    full_name: str | None
    avatar_url: str | None
    bio: str | None
    is_verified: bool
    is_active: bool
    role: str
    created_at: datetime

    model_config = ConfigDict(from_attributes=True)
    # from_attributes=True: Pydantic reads directly from SQLAlchemy ORM instances
    # Fields intentionally excluded: hashed_password, is_locked,
    # failed_login_attempts, locked_until, deleted_at


class ProfileResponse(BaseModel):
    id: int
    full_name: str | None
    avatar_url: str | None
    bio: str | None

    model_config = ConfigDict(from_attributes=True)
```

## 8. Security Design Decisions

### Why refresh tokens are cookies, not JSON

Tokens stored in `localStorage` or `sessionStorage` are accessible to any JavaScript running on the page. A single XSS vulnerability exposes every user's token. An `HttpOnly` cookie is invisible to JavaScript — the browser manages it transparently. The access token (short-lived, in-memory on the client) is the only token JS ever sees.

### Timing-safe login

If a non-existent email is queried, the service still calls `verify_password()` against a pre-computed dummy hash (`_DUMMY_HASH`). This ensures response time is identical whether the email exists or not. Without this, an attacker can enumerate registered emails purely by measuring response latency.

### Token rotation on every refresh

Every call to `/auth/refresh` issues a new refresh token and immediately revokes the old one. If an attacker steals a refresh token and the legitimate user refreshes first, the stolen token is dead before it can be used.

### Token family theft detection

When a user logs in, a `family_id` UUID is created. Every refresh in that session chain keeps the same `family_id`. If a client presents a JWT whose `jti` has already been rotated away (meaning someone replayed an old token), the service detects it as token theft and revokes the **entire family** — terminating all sessions in that login chain and forcing a full re-login.

### Brute force lockout

After `MAX_FAILED_LOGIN_ATTEMPTS` (default: 5) consecutive failures:
- `is_locked = true`, `locked_until = now() + ACCOUNT_LOCKOUT_MINUTES` (default: 15 min)
- The `locked_until` timestamp is embedded in the error message so clients can display a meaningful countdown
- On the next successful login: `failed_login_attempts` resets to `0`, `is_locked` resets to `false`

### Soft delete

`User` rows are never physically deleted. `deleted_at` is set instead. This:
- Enables admin-driven account recovery within a grace period
- Preserves audit trails — who did what, including for deleted accounts
- Maintains referential integrity for related records (todos, etc.)
- Blocks re-registration with the same email (treat deleted email as permanently taken)

### Password change invalidates all sessions

Changing a password signals a potential security event — the old password may have been compromised. All refresh tokens are revoked immediately. Every other device must re-authenticate. The current access token (Bearer) expires naturally within its 15-minute window — revoking individual access tokens is not supported by design (stateless tokens).
